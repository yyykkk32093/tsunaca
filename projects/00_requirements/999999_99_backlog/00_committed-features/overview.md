## CF-1 公開アクティビティ検索

- **分類**: 新機能
- **由来**: wave3 Phase 3 ビジター機能検討時。登録済みビジターのUI動線確保の前提として必要

### 背景
登録済みビジター（アプリ登録済みだがコミュニティ非メンバーのユーザー）をスケジュールに参加させるには、まず**公開アクティビティを発見できる動線**が必要。
現状はコミュニティ検索のみで、アクティビティ単位の検索・閲覧機能がない。

### 想定スコープ
1. **アクティビティ検索API**: 公開コミュニティのアクティビティを検索（キーワード・日付・エリア等）
2. **検索結果画面**: アクティビティ一覧表示（コミュニティ名・日時・場所・参加費等）
3. **アクティビティ詳細閲覧**: 非メンバーでも公開アクティビティの詳細を閲覧可能
4. **ビジター参加申請**: 詳細画面から登録済みビジターとしての参加フロー

### 関連タスク
- wave3 Phase 3 W3-13a（登録済みビジター追加 UseCase / API）が先行実装済みであることが前提
- コミュニティ検索の詳細検索（レビュー要件）との統合も検討

## CF-2 繰り返しスケジュール一括変更

- **分類**: 機能拡張
- **由来**: wave3 Phase 3 追加修正。繰り返しスケジュール生成リアーキテクチャ時に将来機能として分離

### 背景
現在、繰り返しスケジュールの個別編集（時刻変更、場所変更等）は1件ずつ行う必要がある。Google カレンダーのような「この回のみ / この回以降 / すべて」の更新パターンを導入したい。

### 想定スコープ
1. **更新パターン選択UI**: スケジュール編集時に「この回のみ」「この回以降」「すべての回」を選択
2. **「この回のみ」**: 現行の単体 UpdateSchedule で対応済み
3. **「この回以降」**: 指定日以降の未キャンセルスケジュールを一括更新
4. **「すべての回」**: Activity のデフォルト値自体を変更し、既存の未キャンセルスケジュールにも反映

### 技術的メモ
- `RecurringScheduleGenerator` に `updateSchedulesBulk()` メソッドを追加する想定
- 参加者がいるスケジュールの一括変更時は通知が必要（キャンセルとは異なり変更通知）
- Activity.update のデフォルト属性変更と個別スケジュール変更は別の操作として扱う

## CF-3 APIバリデーション基盤整備（全エンドポイント Zodミドルウェア）

- **分類**: 基盤改善
- **由来**: wave4 Phase 3 通知metadata実装時にBEへZod導入。現状は metadata バリデーション限定使用のため、全エンドポイントへの展開を将来機能としてストック

### 背景
現在、APIリクエストバリデーションは各ルートハンドラ内で個別に `req.body as XXX` のような型アサーションを行っている。Zodスキーマを用いたバリデーションミドルウェアを導入することで、入力バリデーションを一元化・型安全化できる。

### 想定スコープ
1. **共通Zodバリデーションミドルウェア**: `validateBody(schema)`, `validateQuery(schema)`, `validateParams(schema)` ミドルウェアを新設
2. **エンドポイントごとのZodスキーマ定義**: 各APIルートの入力スキーマを `*.schema.ts` に定義
3. **エラーレスポンスの統一**: バリデーションエラー時に `{ code: 'VALIDATION_ERROR', errors: [...] }` 形式で返却
4. **段階的移行**: 新規エンドポイントから必須適用、既存エンドポイントは段階的に移行

### 技術的メモ
- BE Zod は v4.3.6 で導入済み（`backend/package.json`）
- 設計方針は `backend/docs/validation/validation-strategy.md` に記載済み
- Express ミドルウェアとして `(req, res, next)` パターンで実装
- UseCase 層の入力型と Zod スキーマの整合性を保つ仕組みが必要（`z.infer<typeof schema>` の活用）

## CF-4 ValueObjectカバレッジ拡充（Fee, ImageUrl 等）

- **分類**: リファクタ
- **由来**: wave4 Phase 3 実装レビュー時。ドメイン層のプリミティブ値が型安全でない箇所が多数存在

### 背景
現在、金額（fee）、画像URL（avatarUrl, coverUrl, logoUrl）、表示名（displayName）等がドメイン層で `string` や `number` のプリミティブ型で扱われている。これらをValueObjectに昇格させることで、バリデーションロジックの集約・不正値の流入防止が実現できる。

### 想定スコープ
1. **Fee / Money VO**: 金額の非負制約、通貨単位（将来の多通貨対応への布石）
2. **ImageUrl VO**: URL形式バリデーション、許可ドメイン制限（S3ドメインのみ等）
3. **DisplayName VO**: 文字数制約（1〜50文字）、禁止文字チェック
4. **Email VO**: メール形式バリデーション（現状 `string` で扱っている箇所の置き換え）
5. **段階的導入**: 新規コード・変更箇所から適用、既存コードは徐々にリファクタ

### 技術的メモ
- 既存の VO パターン: `backend/src/domains/_sharedDomains/` に基盤あり
- エンティティのコンストラクタでVOを受け取り、リポジトリ層でプリミティブに変換するパターンを踏襲
- Prisma との境界（Infrastructure層）で VO ↔ プリミティブの変換を行う

## CF-5 OSM開催場所マスタ最新化ジョブ

- **分類**: 基盤・運用
- **由来**: wave6 Phase 6（W6-07）開催場所入力UX改善。Placeマスタの初期生成は本フェーズで実装するが、定期最新化ジョブの本実装はスコープ外として分離

### 背景
wave6 Phase 6 で OSM (OpenStreetMap) 由来の開催場所マスタ（Place）を導入する。OSM データは継続的に更新されるため、初期投入後も定期的に最新スナップショットで再取得・差分反映しないと、新規施設・改名・閉鎖が反映されず候補品質が劣化する。本フェーズで配置するスケルトン（`backend/src/job/place-osm-refresh/`）と OSM スクリプト一式（`backend/infra/osm/`）を呼び出す形で本実装する。

### 想定スコープ
1. **月次バッチジョブ**: 毎月1日 03:00 JST 実行（cron / EC2 systemd timer 等）
2. **OSMスクリプト連携**: `backend/infra/osm/` の `osm:run-all` を呼び出し、Place を upsert
3. **論理削除制御**: 当回のスナップショットに含まれない既存Placeは `isActive=false` に降格（物理削除はしない、`usageCount > 0` の参照済みPlaceは履歴保持）
4. **blue/green切替**: 一時テーブル/staging schema にロード → 検証クエリ通過後に本番テーブルへ切替（中断時の復旧容易性確保）
5. **監視・通知**: 失敗時にSlack通知、件数異常検知（前回比 ±X% 超で警告）
6. **運用Runbook**: 失敗時の手動再実行手順、PBF取得失敗時の前回スナップショット使い回しポリシー

### 技術的メモ
- スケルトン配置先: `backend/src/job/place-osm-refresh/`（wave6 Phase 6 で骨格作成済み想定）
- スクリプト本体: `backend/infra/osm/`（wave6 Phase 6 で実装済み想定）
- 中間ファイル（`*.pbf` / `*.jsonl`）の保管期間とディスク容量計画が必要（数GB/月 × N世代）
- 既存PlaceとのID連続性: `(source, sourceId)` UNIQUE で安定マッピング、Activity から参照されたPlaceの `usageCount` は保持
- 月次サイクルのため、OSMの最新化遅延は許容（地点鮮度はビジネス要件と合致）

### 関連
- wave6 Phase 6 W6-07: Placeマスタ基盤・初期インポート実装
