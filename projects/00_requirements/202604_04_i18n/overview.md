# 多言語対応（i18n） — 全体プランニング

> 作成日: 2026-03-07
> 案件No: 202603_06
> アプリケーション全体の多言語対応基盤を導入し、日本語以外の言語（英語等）をサポート可能にする

---

## 📐 背景・目的

- 現状、フロントエンド・バックエンドともに全テキストが**日本語ハードコード**
- i18n ライブラリは**完全未導入**（react-i18next, react-intl 等なし）
- 202603_02（バグ修正）では「一旦全部日本語にする」のみ対応し、多言語対応は本案件に切り出し

---

## 📊 現状分析

| レイヤ                        | 現状                                                                         | 対応必要箇所                           |
| ----------------------------- | ---------------------------------------------------------------------------- | -------------------------------------- |
| フロントエンド UI             | 日本語ハードコード（全画面・全コンポーネント）                               | 全 `.tsx` ファイルの文字列リテラル     |
| バックエンド エラーメッセージ | 日本語ハードコード（例: `'キャンセルされたスケジュールには参加できません'`） | UseCase, Domain のエラーメッセージ     |
| バックエンド 通知テキスト     | 日本語ハードコード（例: `'キャンセル待ちから繰り上がりました'`）             | 通知生成ロジック                       |
| DB マスタデータ               | 日本語（カテゴリ名等）                                                       | マスタテーブル or フロント側マッピング |

---

## 🛠️ 技術選定（案）

| 項目             | 候補                                        | 推奨              | 理由                                                                       |
| ---------------- | ------------------------------------------- | ----------------- | -------------------------------------------------------------------------- |
| フロントエンド   | `react-i18next` / `react-intl` (FormatJS)   | **react-i18next** | エコシステム最大、TypeScript対応良好、namespace分割でfeature単位管理可能   |
| 翻訳ファイル形式 | JSON / YAML / ICU MessageFormat             | **JSON**          | react-i18next のデフォルト。ネスト構造でfeature単位に分割                  |
| バックエンド     | キーベース（フロントで翻訳） / サーバ側翻訳 | **キーベース**    | エラーコードを返し、フロント側で翻訳。通知テキストはサーバ側でロケール対応 |
| 言語切替         | ブラウザ設定自動検出 + 手動切替             | **両方**          | `navigator.language` で初期値、設定画面で変更可能                          |

## ⚠️ 前提・依存

- 202603_02（バグ修正・リファクタリング）の完了後に着手が望ましい（UIが安定してからキー化する方が手戻りが少ない）

### 💴 通貨フォーマットに関する注意

現在、参加費の表示は `¥${amount.toLocaleString()}` でハードコードされている（例: `¥1,500`）。
i18n 導入時に以下を対応する必要がある:

- `Intl.NumberFormat` で通貨コード（`JPY` / `USD` 等）をロケールに応じて切替
- 参加費入力フォームのプレースホルダ・バリデーションもロケール対応
- 対象箇所: `ActivityDetailPage`, `ScheduleCard`, `ScheduleDetailPage`, `ParticipationActionButton` 等

---

### 🔧 バリデーションエラーのエラーコード方式への移行

現在、BE Zod バリデーションエラーがデフォルトの英語メッセージ（例: `Too big: expected number to be <=100000`）のまま FE に返却・表示されている。
Phase 2 のエラーメッセージキー化（2-1）と合わせて、以下の構成に移行する:

- **BE**: Zod / ドメインエラーからエラーコード + パラメータを返す（例: `{ code: "FEE_TOO_LARGE", field: "participationFee", params: { max: 100000 } }`）
- **FE**: エラーコードから翻訳キーを引いてユーザー向けメッセージを生成
- **対象**: Activity/Schedule 作成・更新 API のバリデーションエラー全般、ドメインエラー（DomainValidationError）

> 由来: 202603_XX wave4 — 参加費上限（100,000円）実装時に、BE Zod エラーがそのまま表示される問題を確認。FE で即時バリデーション（onChange）は実装済みだが、BE エラー表記はi18n スコープで対応する方針。
